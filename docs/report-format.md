## Overview

Each pipeline run produces a self-contained research report in two formats.
The report is the primary deliverable of the demo — it makes the pipeline's
work tangible and surfaces the gateway's attribution data in a readable form.

Reports are written to `LCLG_OUTPUT_DIR` (default: `./output/`).

---

## Output Files

```
output/
└── <run_id>/
    ├── pipeline_result.json   # Full pipeline output (JSON cache for --render-only)
    ├── report.html            # Self-contained HTML, no external dependencies
    └── report.md              # Markdown, renders cleanly on GitHub
```

Both files contain the same content. The HTML version is the primary artifact;
the Markdown version exists for GitHub rendering and easy sharing in PRs or
documentation.

---

## HTML Report Sections

The HTML report is generated from `src/lclg/report/templates/report.html.jinja2`
using Jinja2. It is fully self-contained — all CSS is embedded, Mermaid diagrams
are rendered via an inline CDN script tag, and no external stylesheets or fonts
are required.

### 1. Header

| Field | Source |
|-------|--------|
| Topic | CLI input / `LCLG_TOPIC` |
| Run ID | UUID generated at pipeline start |
| Timestamp | Pipeline start time (UTC) |
| Total cost | Sum of `metering.usd_charged` across all agent calls |
| Total latency | Wall-clock time from first agent call to ReportAgent completion |
| Mode | `LCLG_MODE` value |

### 2. Pipeline Diagram

A Mermaid flowchart rendered inline showing the actual execution path: which
agents ran, which providers were called, and which sub-questions were researched.
This is generated dynamically from the `PipelineResult` — not a static diagram.

### 3. Research Findings

One accordion panel per sub-question generated by the `PlannerAgent`. Each panel
shows:
- The sub-question
- The `ResearchAgent` response
- Token count and cost for that research call

### 4. Analysis Synthesis

The full narrative output from the `AnalysisAgent`, rendered as formatted text.

### 5. Provider Comparison Table

The centrepiece of the report — the `ComparatorAgent` result shown side by side:

| Field | OpenAI (`gpt-4o`) | Anthropic (`claude-sonnet-4-6`) | Gemini (`gemini-2.5-flash`) |
|-------|-------------------|---------------------------------|-----------------------------|
| Response | ... | ... | ... |
| Latency (ms) | ... | ... | ... |
| Input tokens | ... | ... | ... |
| Output tokens | ... | ... | ... |
| Cost (USD) | ... | ... | ... |

### 6. Final Report

The executive summary produced by the `ReportAgent`, rendered as formatted
sections with headings.

### 7. Cost Summary

A rolled-up view of cost for the run, shown before the per-call detail so
the totals are visible without scrolling through the full breakdown.

**By agent:**

| Agent | Calls | Tokens In | Tokens Out | Cost (USD) |
|-------|-------|-----------|------------|------------|
| PlannerAgent | 1 | ... | ... | ... |
| ResearchAgent | N | ... | ... | ... |
| AnalysisAgent | 1 | ... | ... | ... |
| ComparatorAgent | 3 | ... | ... | ... |
| ReportAgent | 1 | ... | ... | ... |
| **Total** | | | | **$X.XXXXX** |

The grand total must match the **Total cost** shown in the [Header](#1-header)
and the Attribution Breakdown total. The by-agent view is derived from the same
`AgentCall` records that feed the Attribution Breakdown.

> **Not yet implemented:** A "By provider" rollup (OpenAI total, Anthropic total,
> etc.) is planned but not present in the current report. The Attribution
> Breakdown table contains provider-level data and can be sorted/filtered manually.

### 8. Attribution Breakdown

A full per-call audit table of every gateway request made during the run.
The final row is a totals summary.

| Agent | Model | Provider | Workload | Tokens In | Tokens Out | Cost (USD) | Record ID | Decision |
|-------|-------|----------|----------|-----------|------------|------------|-----------|----------|
| planner | gpt-4o-mini | openai | wl_lclg_planner | ... | ... | ... | `rec-...` | allow |
| researcher | claude-haiku-4-5 | anthropic | wl_lclg_researcher | ... | ... | ... | `rec-...` | allow |
| ... | | | | | | | | |
| **Total** | | | | **N** | **N** | **$X.XXXXX** | | |

This table is generated from the `record_id` and `metering` fields returned
by the gateway on each call. It directly demonstrates Axemere's per-call
attribution capability. The totals row must equal the Cost Summary grand total
and the Header total cost — any discrepancy indicates a bug in `builder.py`.

## Markdown Report Sections

The Markdown report mirrors the HTML structure with ATX headings. Mermaid
fenced code blocks are used for the pipeline diagram (renders on GitHub).
The provider comparison and attribution tables use standard Markdown table
syntax.

---

## Design Principles

- **Self-contained** — the HTML file can be emailed, posted to a website, or
  dropped into a PR as a single attachment with no dependencies
- **No external CDN at render time** — CSS is embedded; only the Mermaid CDN
  script is loaded (and only when the HTML is opened in a browser)
- **Attribution-first** — the attribution breakdown section is always present,
  even if individual calls had zero cost, to demonstrate the gateway's recording
  capability
- **Run-specific** — every report includes the run ID, mode, and timestamp so
  multiple reports from the same topic can be compared
